## 1. Purpose and Scope

This document is the low-level design for the **Goose Agent Framework v1**. It describes:

1. The Goose primitives we rely on (plugins, extensions, agents, skills, recipes, hooks, sessions, tools).
2. What the framework *adds* on top of Goose to meet the minion requirements: orchestration, governance, observability, durability, and chat ingress.
3. How the added components interact at runtime.
4. The concrete interfaces, data models, and file layouts a contributor must implement.

The intended reader is an engineer who will build the framework. For the executable plan, see [docs/execplan/execution-plan.md](./execplan/execution-plan.md).

## 2. Goose Foundations

Goose is the agent runtime. It provides the agent loop, tool calling, sessions, and the packaging model. The framework is *not* a replacement for Goose; it is a **Goose plugin plus a set of MCP extensions** that turn Goose into a multi-agent engineering assistant.

### 2.1 Plugin Model

A Goose **plugin** is a git-installable bundle of:

| Asset | Purpose | File Layout |
|---|---|---|
| **Agents** | System prompts that define a role, tools, and output schema | `agents/<name>.md` |
| **Skills** | Reusable instructions the agent can `load` into context | `skills/<name>/SKILL.md` |
| **Commands / Recipes** | Parameterized task templates | `commands/<name>.yaml` |
| **Rules** | Governance, allowlists, model tiers, rate limits | `rules/*.yaml` |
| **Hooks** | Lifecycle scripts triggered by session events | `hooks/hooks.json` + scripts |
| **Manifest** | Plugin metadata and asset paths | `.plugin/plugin.json` |

Installation: `goose plugin install <git-url>` or `goose plugin install file://$(pwd)`.

Plugins are **Markdown/JSON only**. They cannot ship compiled code or MCP servers. This is a hard constraint that shapes the framework's packaging.

### 2.2 Extension Model

A Goose **extension** is an MCP server that exposes tools. Extensions are loaded by:

- CLI flags: `--with-extension "node ./dist/index.js"`, `--with-streamable-http-extension <url>`, `--with-builtin <name>`.
- Config entries in `~/.config/goose/config.yaml`.
- Deeplinks generated by `goose recipe deeplink`.

Extensions provide tools; the agent loop decides when to call them. The framework ships four MCP extensions:

1. `mcp-toolshed` — governed proxy to external MCP servers.
2. `slack-bot` — Slack ingress/egress.
3. `teams-bot` — Teams ingress/egress.
4. `agent-dashboard` — dashboard backend.

### 2.3 Built-In Extensions Used by the Framework

Goose ships built-in extensions. The framework uses these rather than reimplementing them.

| Built-in | Tools / Role | Default State | Framework Usage |
|---|---|---|---|
| `Summon (delegation)` | `delegate`, `load` | Enabled | Spawn minions and load skills/resources into context. |
| `Developer` | `shell`, `edit`, `write`, `tree`, file read | Enabled | Filesystem and shell access for minions. |
| `Analyze` | `analyze` | Enabled | Tree-sitter code analysis. |
| `Apps` | `apps__create_app`, `apps__iterate_app`, `apps__list_apps`, `apps__delete_app` | Enabled | Build the operator dashboard UI. |
| `Todo` | `todo__todo_write` | Enabled | Optional task tracking in long sessions. |
| `Extension Manager` | `extensionmanager__*` | Enabled | List/manage extensions at runtime (diagnostics only). |
| `Chat Recall` | Session memory lookup | **Disabled** | Load prior context into the orchestrator. Must enable in config. |
| `Orchestrator` | `list_sessions`, `view_session`, `interrupt_agent` | **Disabled** | Monitor and cancel running minions. Must enable in config. |

Important: in Goose 1.37.0, `--with-builtin <name>` adds an extension to a run but **does not override a disabled config entry**. `chatrecall` and `orchestrator` must be enabled in `~/.config/goose/config.yaml` (or Goose must be started with `--no-profile`).

### 2.4 Agent Loop and Tools

Goose runs an agent loop:

1. Receive a user message.
2. Add it to the session context.
3. Ask the configured LLM what to do next.
4. The LLM may emit tool calls.
5. Goose executes the tools and returns results to the LLM.
6. Repeat until the LLM returns a final answer.

A tool is a JSON-schema-defined function. Tool names are namespaced by extension, e.g.:

- `shell` (Developer)
- `analyze` (Analyze)
- `delegate` (Summon)
- `load` (Summon)
- `execute_tool` (mcp-toolshed)
- `apps__create_app` (Apps)
- `orchestrator__list_sessions` (Orchestrator)

### 2.5 `delegate` vs `start_agent`

Goose provides two ways to spawn sub-agents:

| Mechanism | Inherits Parent Extensions? | Tool Access | Framework Usage |
|---|---|---|---|
| `delegate` | **Yes** | Gets Developer, Analyze, mcp-toolshed, etc. | **All minion dispatch.** |
| `start_agent` (Orchestrator) | No | Bare agent with no tools | Do not use for minions. Use only for monitor/cancel via `list_sessions`, `view_session`, `interrupt_agent`. |

Live testing confirmed that a `start_agent` sub-agent could not run `ls` because it had no shell tools, while a `delegate(async: true)` sub-agent succeeded.

### 2.6 Delegate Tool Schema

The exact `delegate` tool signature (Goose 1.37.0) is:

```json
{
  "name": "delegate",
  "description": "Delegate a task to a subagent that runs independently with its own context. ...",
  "parameters": {
    "type": "object",
    "properties": {
      "instructions": {
        "type": "string",
        "description": "Task instructions. Required for ad-hoc tasks."
      },
      "source": {
        "type": "string",
        "description": "Name of a recipe or agent to run."
      },
      "parameters": {
        "type": "object",
        "additionalProperties": true,
        "description": "Parameters for the source (only valid with source)."
      },
      "async": {
        "type": "boolean",
        "description": "Run asynchronously. Collect results with load(source: taskId)."
      },
      "extensions": {
        "type": "array",
        "items": { "type": "string" },
        "description": "Extensions to expose to the delegate. Omit to inherit parent's extensions; [] for bare agent."
      },
      "provider": { "type": "string" },
      "model": { "type": "string" },
      "max_turns": { "type": "number" }
    }
  }
}
```

Invocation modes:

| Mode | `source` | `instructions` | Purpose |
|---|---|---|---|
| Ad-hoc | omitted | required | Generic sub-agent with custom instructions. |
| Agent/recipe source | agent/recipe name | optional override | Run a registered agent or recipe from the plugin. |
| Combined | agent/recipe name | required | Load the agent identity, then add a concrete task. |

Example — dispatch the Code Explorer minion:

```json
{
  "source": "code-explorer",
  "instructions": "Find the code responsible for login timeout. Ticket: INC00421.",
  "async": true,
  "extensions": ["developer", "analyze", "mcp-toolshed"]
}
```

### 2.7 Result Collection with `load`

For `async: true` delegates, Goose returns a task ID. The orchestrator collects results with the `load` tool:

```json
{
  "name": "load",
  "parameters": {
    "source": "<task_id_returned_by_delegate>"
  }
}
```

The orchestrator prompt instructs:

```
Decompose → async delegates → load(source: task_id) for each → synthesize.
```

`load` blocks until the delegate finishes and returns its final output. The orchestrator must keep the mapping of `correlation_id → delegate task_id` in the `SessionStore`.

### 2.8 Agent File Format

Agent files are Markdown with YAML frontmatter. Example `agents/code-explorer.md`:

```markdown
---
name: code-explorer
description: Explores a codebase to answer questions or locate root causes
---

# Code Explorer

## Identity
- **Role:** code-explorer
- **Purpose:** read and search code, never write it
- **Vibe:** methodical, concise

## What I do
- Use `tree` and `analyze` to map the repo
- Read targeted files via `shell`/`developer` tools
- Return structured findings

## What I don't do
- I don't edit or commit code
- I don't run destructive shell commands

## Output format
Return JSON matching `schemas/code-explorer-output.json`.
```

`source` in `delegate` must match the `name` field in the agent frontmatter. The agent's Markdown body becomes the sub-agent's system instructions; the `instructions` parameter adds the concrete task.

### 2.9 Sessions and Memory

A **session** is a single conversation thread. Goose persists sessions locally under `~/.local/share/goose/` and logs to `~/.local/state/goose/logs`.

The framework adds a `SessionStore` (SQLite) to hold:

- Session metadata (session ID, team, platform, user).
- Minion run records (run ID, correlation ID, status, result pointer, token usage).
- Pending approvals (approval ID, requested tool, requester, timeout).
- Tool-call cache (deduplication of identical calls within a session).

`Chat Recall` provides native session memory. The orchestrator loads relevant prior context before dispatching downstream minions.

### 2.10 Recipes / Commands

Recipes are parameterized YAML task templates. They are stored in the plugin's `commands/` directory. Goose does **not** scan plugin `commands/` for recipes automatically. Discovery paths are:

- Current directory (`*.yaml`, `*.json`).
- `$GOOSE_RECIPE_PATH`.
- `~/.config/goose/recipes/`.
- `./.goose/recipes/`.
- `$GOOSE_RECIPE_GITHUB_REPO`.

Therefore plugin recipes are run/validated by full path:

```bash
goose run --recipe ~/.agents/plugins/goose-agent-framework/commands/daily-pr-review.yaml
goose recipe validate ~/.agents/plugins/goose-agent-framework/commands/daily-pr-review.yaml
```

For local development, export `GOOSE_RECIPE_PATH=$HOME/.agents/plugins/goose-agent-framework/commands` so `goose recipe list` finds them.

### 2.11 Hooks

The plugin manifest references `hooks/hooks.json`. Hooks are lifecycle callbacks the Goose runtime can invoke. The framework uses hooks for:

- Session start/end logging.
- Backup triggers (SQLite → Blob Storage).
- Correlation ID seeding.

### 2.12 ACP and `goose serve`

Goose exposes an **Agent Client Protocol (ACP)** server via `goose serve`. ACP is a JSON-RPC 2.0 protocol over WebSocket or streamable HTTP. It is the recommended way for external clients (chat bots, dashboard, IDE plugins) to interact with Goose.

#### Connection Establishment (WebSocket)

```
Client                              Goose Server
  │  GET /acp                        │
  │  Upgrade: websocket              │
  │─────────────────────────────────►│
  │  HTTP 101                        │
  │  Acp-Connection-Id: <uuid>       │
  │◄─────────────────────────────────│
```

After the WebSocket upgrade, all traffic is JSON-RPC 2.0 text frames.

#### Core Methods

| Method | Direction | Purpose |
|---|---|---|
| `initialize` | C→S | Negotiate capabilities, receive `connectionId`. |
| `session/new` | C→S | Create a session. |
| `session/prompt` | C→S | Send a user prompt and start an agent turn. |
| `session/cancel` | C→S | Cancel the in-progress turn. |
| `session/load` | C→S | Resume an existing session (history replay). |
| `session/list` | C→S | List sessions. |
| `session/fork` | C→S | Fork a session at a message. |
| `session/close` | C→S | Close a session. |
| `session/delete` | C→S | Delete a session. |
| `request_permission` | S→C | Server asks the client for human approval. |
| `session/update` | S→C | Usage/cost update notification. |
| `notifications/AgentMessageChunk` | S→C | Streaming response chunk. |
| `notifications/ToolCall` | S→C | Tool invocation started. |
| `notifications/ToolCallUpdate` | S→C | Tool result/status update. |

#### Example: Start a Session and Prompt

**Client → Server:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "session/new",
  "params": {
    "sessionId": null,
    "cwd": "/Volumes/ExtDisk1/Minions"
  }
}
```

**Server → Client:**

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { "sessionId": "sess_abc123" }
}
```

**Client → Server:**

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "session/prompt",
  "params": {
    "sessionId": "sess_abc123",
    "prompt": [
      { "type": "user", "text": "@minions review PR #342" }
    ]
  }
}
```

**Server → Client** (streaming):

```json
{ "jsonrpc": "2.0", "method": "notifications/AgentMessageChunk", "params": { "sessionId": "sess_abc123", "text": "Reviewing..." } }
{ "jsonrpc": "2.0", "method": "notifications/ToolCall", "params": { "sessionId": "sess_abc123", "toolCallId": "call_1", "name": "execute_tool", "arguments": { ... } } }
{ "jsonrpc": "2.0", "method": "notifications/ToolCallUpdate", "params": { "sessionId": "sess_abc123", "toolCallId": "call_1", "status": "completed", "output": [...] } }
{ "jsonrpc": "2.0", "id": 2, "result": { "sessionId": "sess_abc123", "stopReason": "end_turn" } }
```

#### Permission Requests

ACP has a built-in human-in-the-loop mechanism. When the agent attempts a destructive tool, Goose sends:

```json
{
  "jsonrpc": "2.0",
  "id": 99,
  "method": "request_permission",
  "params": {
    "sessionId": "sess_abc123",
    "toolCallId": "call_2",
    "toolName": "execute_tool",
    "arguments": { "server_alias": "github", "tool_name": "merge_pull_request", ... },
    "confirmationDetails": {
      "title": "Destructive operation",
      "message": "This will merge PR #342.",
      "options": [
        { "id": "allow_once", "label": "Allow once" },
        { "id": "allow_always", "label": "Always allow" },
        { "id": "reject_once", "label": "Reject" }
      ]
    }
  }
}
```

The client (Slack/Teams/dashboard) renders the prompt and returns the user's choice.

#### Running `goose serve`

```bash
GOOSE_SERVER__SECRET_KEY='my-secret' goose serve --host 0.0.0.0 --port 3284
```

Clients connect to:

- `ws://localhost:3284/acp?token=my-secret` (WebSocket)
- `POST http://localhost:3284/acp` (JSON-RPC over HTTP)
- `GET http://localhost:3284/acp` (SSE stream)

#### ACP vs MCP

| Concern | MCP | ACP |
|---|---|---|
| Transport | SSE (per request) or stdio | Long-lived WebSocket or streamable HTTP |
| `initialize` | Returns SSE stream | Returns `200 OK` JSON |
| Session lifecycle | None native | `session/new`, `session/load`, `session/close`, `session/delete` |
| Human approval | Extension-specific | Built-in `request_permission` |
| Usage tracking | Extension-specific | Built-in `session/update` |
| Identity | `Mcp-Session-Id` | `Acp-Connection-Id` + `Acp-Session-Id` |

#### Framework Usage of ACP

- **Slack Bot** and **Teams Bot** connect to `goose serve` via ACP WebSocket. They translate chat messages into `session/prompt` calls and render ACP notifications as Slack/Teams messages.
- **Agent Dashboard** connects via ACP to list sessions, view transcripts, and send `session/cancel`.
- **External ACP clients** (IDEs, CLI clients) can connect directly to the framework's Goose server and load the framework plugin.

## 3. Framework Additions — Overview

The framework layers the following capabilities on top of Goose:

| Concern | Goose Primitive | Framework Addition |
|---|---|---|
| Packaging | Plugin + extensions | Framework plugin + 4 MCP extensions |
| Orchestration | `delegate` tool | Orchestrator agent + DAG engine + result collector |
| Minion specialization | Agent prompts | Five typed minion agents with output schemas |
| Tool governance | Extension list | MCP toolshed with allowlists, path scoping, rate limits, circuit breakers |
| Chat ingress | `goose serve` / custom code | Slack-bot and Teams-bot MCP extensions |
| State | Session files | `SessionStore` (SQLite) + Azure Table/Blob |
| Async durability | `delegate(async: true)` | Azure Service Bus queue for complex pipelines |
| Human approval | Agent prompt | Approval engine backed by SQLite + chat buttons |
| Observability | Logs | Hierarchical correlation IDs + tool-call log + Log Analytics/Grafana |
| Cost control | Model config | Per-minion token budgets and model-tier routing |

## 4. Plugin Design

### 4.1 Plugin Manifest

`.plugin/plugin.json`:

```json
{
  "name": "goose-agent-framework",
  "version": "0.1.0",
  "description": "Multi-agent orchestration framework for engineering operations.",
  "repository": "https://github.com/your-org/goose-agent-framework",
  "agents": "./agents/",
  "skills": "./skills/",
  "commands": "./commands/",
  "rules": "./rules/",
  "hooks": "./hooks/hooks.json"
}
```

### 4.2 Agents

Agents are Markdown files with YAML frontmatter.

`agents/orchestrator.md`:

```yaml
---
name: orchestrator
description: Routes user requests, builds a DAG of minions, and synthesizes results
model_tier: reasoning
token_budget: 50000
allowed_skills:
  - intent-classification
  - task-decomposition
  - result-synthesis
  - delegate-management
allowed_extensions:
  - developer
  - analyze
  - chatrecall
  - orchestrator
  - mcp-toolshed
---
# Orchestrator

## Identity
- **Role:** orchestrator
- **Purpose:** hold the conversation thread and route work to minions
- **Vibe:** warm, direct, no filler

## What I do
- Classify intent and decompose complex work into a DAG
- Dispatch minions via the `delegate` tool
- Collect results via `load(source: task_id)`
- Synthesize responses back to the user

## What I don't do
- I don't do deep code work → delegate to code-explorer / code-reviewer / pr-crafter
- I don't fetch ticket details → delegate to ticket-analyst
- I don't run security audits → delegate to security-auditor
```

Minion agents follow the same pattern:

| Agent File | `name` | `minion_type` | Model Tier | Primary Tools |
|---|---|---|---|---|
| `code-explorer.md` | `code-explorer` | `code_explorer` | reasoning | `shell`, `tree`, `analyze`, `execute_tool(git)`, `execute_tool(ado)` |
| `code-reviewer.md` | `code-reviewer` | `code_review` | code_review | `shell`, `analyze`, `execute_tool(github)`, `execute_tool(ado)` |
| `pr-crafter.md` | `pr-crafter` | `pr_create` | code_generation | `shell`, `edit`, `write`, `execute_tool(github)`, `execute_tool(ado)` |
| `ticket-analyst.md` | `ticket-analyst` | `ticket_lookup` | fast | `execute_tool(servicenow)`, `execute_tool(jira)`, `execute_tool(ado)` |
| `security-auditor.md` | `security-auditor` | `security_audit` | security | `shell`, `analyze`, `execute_tool(github)`, `execute_tool(ado)` |

### 4.3 Skills

Skills are reusable prompt modules loaded via the Summon `load` tool.

`skills/intent-classification/SKILL.md`:

```markdown
# Intent Classification

Classify the user message into one of:
- `ticket_lookup`
- `ticket_summary`
- `code_explore`
- `code_review`
- `pr_create`
- `security_audit`
- `ticket_fix_pr`

Return JSON matching `schemas/intent.json`.
```

The orchestrator uses `load` to pull skills into its context before classification, decomposition, or synthesis.

### 4.4 Commands / Recipes

Recipes live in `commands/` and are validated with `goose recipe validate`. Example `commands/daily-pr-review.yaml`:

```yaml

parameters:
  - key: repositories
    description: Comma-separated list of repos
    input_type: text
    requirement: required
instructions: |
  Use the code-reviewer minion to review each open PR in {{repositories}}.
  Post a summary to the channel.
```

### 4.5 Rules

`rules/` contains governance files:

- `rules/allowlists.yaml` — per-minion tool allowlists.
- `rules/governance.yaml` — destructive actions requiring approval, rate-limit defaults, workspace boundaries.
- `rules/models.yaml` — tier-to-deployment mapping and per-minion token budgets.

### 4.6 Hooks

`hooks/hooks.json` registers scripts for session lifecycle events. Example:

```json
{
  "on_session_start": ["./hooks/log_session_start.sh"],
  "on_session_end": ["./hooks/backup_sqlite.sh"]
}
```

## 5. MCP Toolshed Extension

The toolshed is the single interception point for every external tool call. It is an MCP server that wraps other MCP servers.

### 5.1 Responsibilities

1. **Server registration** — maintain a catalog of MCP server aliases and connection parameters.
2. **Allowlist enforcement** — reject calls not permitted for the requesting minion type.
3. **Path scoping** — restrict filesystem/git operations to allowed paths (ADR-019).
4. **Rate limiting** — throttle per-server, per-minion, and per-team.
5. **Circuit breaker** — fail fast when a server is unhealthy.
6. **Audit logging** — write every call to Azure Table Storage with correlation ID.
7. **Tool-call caching** — deduplicate identical calls within a session.
8. **Human approval gating** — pause destructive calls until a human approves.

### 5.2 Tool Registration

`extensions/mcp-toolshed/config/servers.yaml`:

```yaml
servers:
  github:
    alias: github
    transport: stdio
    cmd: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "${KEYVAULT_GITHUB_TOKEN}"
  ado:
    alias: ado
    transport: sse
    url: https://ado-mcp.internal/mcp
```

The toolshed spawns or connects to each registered server and exposes a unified `execute_tool` interface.

### 5.3 Exposed Tool: `execute_tool`

```json
{
  "name": "execute_tool",
  "description": "Execute a tool on behalf of a minion through the governed toolshed.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "correlation_id": { "type": "string" },
      "team_id": { "type": "string" },
      "minion_type": { "type": "string" },
      "server_alias": { "type": "string" },
      "tool_name": { "type": "string" },
      "params": { "type": "object" }
    },
    "required": ["correlation_id", "minion_type", "server_alias", "tool_name"]
  }
}
```

### 5.4 Allowlist Enforcement

`rules/allowlists.yaml`:

```yaml
allowlists:
  code_explorer:
    github: ["search_repositories", "get_file_contents", "list_commits"]
    ado: ["search_code", "get_item"]
    filesystem: ["read_file", "list_directory"]
  pr_crafter:
    github: ["create_branch", "create_or_update_file", "create_pull_request"]
    filesystem: ["read_file", "write_file", "edit_file"]
```

If a minion calls a tool outside its allowlist, the toolshed returns:

```json
{
  "status": "blocked_by_allowlist",
  "error": "Tool 'delete_branch' is not allowed for minion_type 'code_explorer'",
  "correlation_id": "corr_xxx.1.github-003"
}
```

### 5.5 Path Scoping

Filesystem tools are scoped per minion:

```yaml
path_scopes:
  code_explorer:
    mode: allowlist
    paths: ["/repo"]
    deny: [".git/", "node_modules/", "secrets/"]
```

### 5.6 Rate Limiting

`extensions/mcp-toolshed/config/rate-limits.yaml`:

```yaml
rate_limits:
  github:
    requests_per_minute: 30
    burst: 10
  servicenow:
    requests_per_minute: 60
    burst: 20
```

Exceeded limits return `status: throttled` with `retry_after_seconds`.

### 5.7 Circuit Breaker

State machine per server alias (ADR error-handling):

- `closed` → `open` after `failure_threshold` consecutive failures.
- `open` → `half-open` after `timeout_secs`.
- `half-open` → `closed` after `success_threshold` consecutive successes.
- Any failure in `half-open` re-opens.

Open breakers return `status: error` with `retry_after_seconds`.

### 5.8 Audit Logging

Every `execute_tool` call writes a row to Azure Table Storage:

| Property | Value |
|---|---|
| PartitionKey | `<team_id>` |
| RowKey | `<correlation_id>.<timestamp>` |
| session_id | UUID |
| minion_type | e.g. `code_explorer` |
| server_alias | e.g. `github` |
| tool_name | e.g. `get_file_contents` |
| params_hash | SHA-256 of params |
| status | success / error / blocked / throttled |
| duration_ms | integer |
| token_usage | integer |

### 5.9 Tool-Call Caching

Identical calls (same server, tool, params) within the same session are cached in SQLite for the session lifetime. The cache key is `sha256(server_alias + tool_name + canonical_params + session_id)`.

### 5.10 Human Approval Gating

Destructive tools (merge, close ticket, deploy) require approval. The toolshed:

1. Detects the tool in `governance.yaml` destructive list.
2. Writes a `PendingApproval` record to SQLite.
3. Returns `status: approval_required` with an `approval_id`.
4. Posts an approval card to Slack/Teams.
5. Waits for the user response.
6. On approval, executes the tool and logs it; on denial, returns `status: denied`.

See [Interfaces and Dependencies](#12-interfaces-and-dependencies) for TypeScript interfaces.

## 6. Orchestrator Runtime Flow

The orchestrator is a Goose agent. It is not a separate service. Its prompt instructs Goose to use the Summon `delegate` tool to dispatch minions and the built-in `Orchestrator` extension to monitor them.

### 6.1 Request Intake

A request arrives from one of:

- Slack Bot MCP extension (`slack-bot`).
- Teams Bot MCP extension (`teams-bot`).
- A recipe run (`goose run --recipe commands/daily-pr-review.yaml`).
- A Service Bus message for an async continuation.

The ingress layer creates a session (or resumes one) and seeds the root correlation ID: `corr_<uuid>`.

### 6.2 Intent Classification

The orchestrator loads the `intent-classification` skill and classifies the request:

```json
{
  "intent": "code_review",
  "complexity": "simple",
  "platform": "slack",
  "entities": { "pr_number": 342 }
}
```

- **Simple intents** are handled by a single minion via `delegate(async: false)`.
- **Complex intents** are decomposed into a DAG and executed via `delegate(async: true)`.

### 6.3 Task Decomposition (DAG)

Complex intents produce a DAG of phases. Example `ticket_fix_pr`:

```
Phase 1: ticket-analyst (fetch ticket details)
Phase 2: code-explorer (find relevant code) [depends on Phase 1]
Phase 3: pr-crafter (create branch and fix) [depends on Phase 2]
         code-reviewer (pre-review)         [depends on Phase 2]
Phase 4: pr-crafter (open PR)               [depends on Phase 3]
```

Independent minions within a phase run in parallel.

### 6.4 Dispatch

The orchestrator calls `delegate` using the agent-as-source pattern. The `source` matches the agent's frontmatter `name`; the `instructions` add the concrete task. Omitting `extensions` lets the delegate inherit the orchestrator's extensions.

```json
{
  "source": "code-explorer",
  "instructions": "Find the code responsible for login timeout. Ticket: INC00421.",
  "async": true
}
```

Goose returns a task ID for `async: true` delegates. The orchestrator immediately records:

```json
{
  "correlation_id": "corr_abc.1",
  "delegate_task_id": "task_xxx",
  "status": "running"
}
```

in the `SessionStore`.

Because `delegate` inherits extensions by default, the sub-agent has access to Developer, Analyze, and the mcp-toolshed. If tighter scoping is required, the orchestrator can pass `extensions: ["developer", "analyze", "mcp-toolshed"]`.

### 6.5 Result Collection

For `async: true` delegates, the orchestrator collects results with the `load` tool:

```json
{
  "source": "task_xxx"
}
```

The orchestrator workflow is:

```
Decompose → async delegates → load(source: task_id) for each → synthesize.
```

Result collection steps:

1. Call `load(source: <delegate_task_id>)` for each finished minion.
2. Store the raw result in `SessionStore` keyed by `correlation_id`.
3. Validate the returned JSON against the minion's output schema.
4. If validation fails, retry once by re-delegating with the invalid output appended as feedback.
5. For stuck or runaway minions, use `orchestrator__view_session` and `orchestrator__interrupt_agent`.

Persistent failures are surfaced to the user with the four-field error template and the relevant correlation ID.

### 6.6 Synthesis and Response

The orchestrator loads the `result-synthesis` skill and merges minion outputs into a platform-agnostic response. Platform adapters (Slack Block Kit, Teams Adaptive Card) render the response for the originating channel.

### 6.7 Token Budget Enforcement

Per-minion budgets are defined in `rules/models.yaml`. The orchestrator checks cumulative token usage after each result. If a minion exceeds 80% of its budget, it is instructed to summarize and return final JSON. If it exceeds 100%, the run is marked `budget_exceeded`.

## 7. Minion Agent Design

### 7.1 Frontmatter

```yaml
---
name: code-explorer
minion_type: code_explorer
model_tier: reasoning
token_budget: 25000
output_schema: "../../schemas/code-explorer-output.json"
---
```

### 7.2 Prompt Structure

Each minion prompt follows the Office-Town agent format:

1. **Frontmatter** — `name`, `description`, `minion_type`, `model_tier`, `token_budget`, `output_schema`, `allowed_extensions`.
2. **Identity** — role, purpose, vibe.
3. **What I do** — concrete responsibilities.
4. **What I don't do** — guardrails and delegation boundaries.
5. **Tool guidance** — which tools are available and how to call `execute_tool`.
6. **Output schema** — the JSON schema the final answer must match.
7. **Correlation ID** — the ID to tag every tool call.
8. **Token budget hint** — "You have 25,000 tokens. Search first, read targeted files, then reason."
9. **Progressive disclosure rule** — "Never read the entire repo. Use tree/search, then read only the files you need."

### 7.3 Output Schemas

Schemas live in `schemas/<minion>.json`. Example `code-explorer-output.json`:

```json
{
  "type": "object",
  "properties": {
    "summary": { "type": "string" },
    "files_examined": { "type": "array", "items": { "type": "string" } },
    "root_cause": { "type": "string" },
    "recommended_next_steps": { "type": "array", "items": { "type": "string" } }
  },
  "required": ["summary", "files_examined"]
}
```

Malformed output triggers a retry with feedback.

### 7.4 Error Template

Every user-facing error uses the four-field template:

```json
{
  "severity": "failure",
  "summary": "Could not fetch PR #342",
  "cause": "GitHub API returned 404",
  "impact": "No review can be generated for this PR.",
  "action": "Check the PR number and try again. Correlation ID: corr_abc.1.github-003"
}
```

Severity levels: `success`, `degraded`, `failure`, `infrastructure`.

## 8. Chat Ingress Extensions

The chat bots are thin adapters between Slack/Teams and the Goose ACP server. They do not contain orchestration logic; they turn chat events into ACP `session/prompt` calls and turn ACP notifications into chat messages.

### 8.1 Slack Bot

`extensions/slack-bot/` uses Slack Bolt and an ACP client library.

- Receives `@minions` mentions and DMs via Slack Events API.
- Verifies Slack signing secret.
- Opens a WebSocket ACP connection to `goose serve`.
- Sends `session/new` (if needed) and `session/prompt` for each message.
- Listens for `notifications/AgentMessageChunk`, `notifications/ToolCall`, and `request_permission`.
- Renders threaded replies with Block Kit.
- Posts approval buttons that translate the user's choice into an ACP permission response.

### 8.2 Teams Bot

`extensions/teams-bot/` uses the Microsoft 365 Agent SDK and an ACP client library.

- Receives `@minions` mentions in channels, group chats, and personal chats.
- Authenticates via Azure AD / Microsoft Entra ID.
- Connects to `goose serve` via ACP WebSocket.
- Renders Adaptive Cards for messages, tool results, and permission requests.
- Supports deep links to Azure DevOps work items.

### 8.3 Shared Platform Formatter

Both bots use a shared response formatter in `packages/framework-core/src/platform-formatter.ts` to convert ACP notifications and the orchestrator's platform-agnostic response into Slack Block Kit or Teams Adaptive Card JSON.

## 10. State and Storage

### 10.1 SessionStore

`packages/framework-core/src/store.ts` defines the abstract store interface. Implementations:

- **SQLite** (`extensions/mcp-toolshed/src/sqlite-store.ts`) — primary session state.
- **Azure Table Storage** — immutable tool-call log.
- **Azure Blob Storage** — large artifacts and SQLite backups.

### 10.2 SQLite Schema

```sql
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  team_id TEXT NOT NULL,
  platform TEXT NOT NULL,
  user_id TEXT NOT NULL,
  correlation_root TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

CREATE TABLE minion_runs (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL REFERENCES sessions(id),
  minion_type TEXT NOT NULL,
  correlation_id TEXT NOT NULL,
  status TEXT NOT NULL,
  result_json TEXT,
  tokens_used INTEGER,
  created_at INTEGER NOT NULL,
  completed_at INTEGER
);

CREATE TABLE pending_approvals (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  correlation_id TEXT NOT NULL,
  server_alias TEXT NOT NULL,
  tool_name TEXT NOT NULL,
  params_json TEXT NOT NULL,
  requested_at INTEGER NOT NULL,
  timeout_at INTEGER NOT NULL,
  decision TEXT,
  decided_at INTEGER
);

CREATE TABLE tool_call_cache (
  session_id TEXT NOT NULL,
  cache_key TEXT NOT NULL,
  value_json TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  PRIMARY KEY (session_id, cache_key)
);
```

### 10.3 Azure Table Storage Layout

Table: `ToolCallLog`

| Field | Source | Purpose |
|---|---|---|
| PartitionKey | `team_id` | Query by team |
| RowKey | `correlation_id#timestamp` | Unique, sortable row |
| SessionId | session ID | Join with SQLite |
| MinionType | minion_type | Filter by minion |
| ServerAlias | server_alias | Filter by backend |
| ToolName | tool_name | Filter by tool |
| ParamsHash | SHA-256 | Deduplication / replay |
| Status | status | Success / error / blocked / throttled |
| DurationMs | duration | Performance |
| TokenUsage | tokens_used | Cost tracking |

### 10.4 Azure Blob Storage Layout

Container: `minion-outputs`
Path: `sessions/{corr_id}/{minion_id}/result.json`

Container: `sqlite-backups`
Path: `orchestrator/{replica}/goose-sessions-{timestamp}.db`

### 10.5 Async Queue

Azure Service Bus queue `minion-tasks` holds async continuations for complex pipelines. Messages include:

```json
{
  "session_id": "...",
  "correlation_id": "corr_abc.2",
  "phase": 3,
  "minion_type": "pr_crafter",
  "task": "...",
  "dependencies": ["corr_abc.1"]
}
```

Service Bus sessions guarantee ordered delivery per correlation root.

## 11. Governance and Security

### 11.1 Allowlists

Per-minion tool allowlists are enforced by the toolshed. No minion can call a tool outside its list.

### 11.2 Path Scoping

Filesystem operations are scoped to approved directories with denylist patterns (`.git/`, `node_modules/`, `secrets/`).

### 11.3 Human-in-the-Loop

Destructive actions defined in `rules/governance.yaml` require approval. Approval requests are posted to the originating chat channel with Approve/Deny buttons. Timeouts default to 15 minutes.

### 11.4 Secrets

Secrets (GitHub PATs, ServiceNow credentials, Teams app secrets) are stored in Azure Key Vault and injected via managed identity. No secrets are committed to the plugin repo.

### 11.5 Multi-Tenancy

`team_id` is threaded through sessions, storage prefixes, and governance lookups. The first deployment uses a single default team. Adding teams is config-only.

### 11.6 Network Security

- Azure Container Apps in a private VNet.
- Private endpoints for Service Bus, Storage, Key Vault, AI Foundry.
- Bot ingress is public HTTPS only; endpoints authenticated by Slack/Teams.

- `tool_name`
- `status`

## 14. Testing Strategy Hooks

The low-level design supports the testing pyramid from `./testing-strategy.md`:

| Level | How |
|---|---|
| Unit | Jest tests for `packages/framework-core` and each extension. **100% coverage required.** |
| Integration | Mock MCP servers; run end-to-end `delegate` flows. |
| Prompt quality | Compare candidate prompts against baselines on test cases. |
| E2E | Run `@minions` commands in a staging Slack/Teams workspace. |
| Performance | k6/Artillery against `goose serve` endpoints. |
| Chaos | Kill orchestrator, block MCP server, exhaust rate limits. |

### Coverage Gate

Every package in `packages/` and `extensions/` must maintain **100% line, branch, function, and statement coverage**. Per-package `jest.config.js` files set `coverageThreshold: { global: { branches: 100, functions: 100, lines: 100, statements: 100 } }`. The CI pipeline runs `pnpm test --coverage` and fails if any package drops below 100%. Exemptions require a written justification and explicit exclusion in the package's Jest config.

### Red Build Policy — "Ralph Wiggum" Loop

The CI pipeline runs `pnpm typecheck`, `pnpm build`, `pnpm lint`, `pnpm test --coverage`, and the 100% coverage gate. A failure in any of these steps blocks merge. The author must fix the root cause, re-run the full pipeline, and obtain maintainer/QA approval. There is no merge-on-red, no threshold-lowering without an ADR-level exemption, and no skipping required checks.

## 16. Summary of What the Framework Adds

| Requirement | Primitive Goose Provides | Framework Addition |
|---|---|---|
| Multi-agent orchestration | `delegate` tool | Orchestrator agent, DAG engine, skills, result synthesis |
| Specialized engineering roles | Agent prompts | Five typed minion agents with output schemas |
| Tool governance | Extension list | MCP toolshed with allowlists, path scoping, rate limits, circuit breakers |
| Chat ingress | ACP (`goose serve`) | Slack-bot and Teams-bot ACP clients |
| Audit trail | Session logs | Azure Table Storage tool-call log + correlation IDs |
| Human approval | Agent prompt | Approval engine + chat buttons |
| Cost control | Model config | Per-minion token budgets and tier routing |
| Durability | SQLite sessions | `SessionStore`, Service Bus async queue, Blob backups |
| Observability | stdout logs | Structured logging, Log Analytics, Grafana dashboards |
| Deployment | Local CLI | Azure Bicep, Container Apps, CI/CD |

The result is a Goose-native multi-agent system that feels like a single `@minions` assistant in chat, but is composed of governed, observable, cost-controlled minions behind the scenes.
